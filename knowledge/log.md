## 2026-05-15 — зафиксирован вопрос ГИПу по подсчёту торцевых колонн

Создана страница `knowledge/wiki/decisions/pending-gip-column-count-question.md`.

На странице описано расхождение между текущей моделью приложения, где `edge/middle` считаются по `interiorFrameCount`, и draft-моделью по всем рамным осям `N`. Для сценария `SCN-BUILDING-COUNT-002` приведён пример разницы: `54` колонны по внутренним рамам против `60` колонн по всем осям, где `6` колонн относятся к торцевым осям.

Решение по классификации торцевых колонн, фахверка и дополнительных осей должно быть получено от ГИПа до изменения рабочей модели количества.
## 2026-05-15 — получен ответ ГИПа по торцевым колоннам

Решение зафиксировано в `knowledge/wiki/decisions/2026-05-15-gip-column-count-answer.md`.

Принято: для здания без крана торцевые колонны считать отдельной группой фахверка/торцевых элементов. Для здания с краном, независимо от типа крана, торцевые колонны считать основными колоннами рам.

Следствие для разработки: будущая модель "Единого здания" должна переключать классификацию торцевых колонн по признаку наличия крана и не допускать двойного счёта фахверка.
## 2026-05-15 — реализован режим подсчёта торцевых колонн по признаку крана

Решение ГИПа из `knowledge/wiki/decisions/2026-05-15-gip-column-count-answer.md` закреплено в helper `src/building/unifiedLayout.ts`.

Добавлен явный признак `hasCrane`: без крана основные колонны считаются по внутренним рамам, а торцы остаются отдельной группой фахверка; с краном основные колонны считаются по всем рамам, а торцевой фахверк для этих колонн обнуляется, чтобы не было двойного счёта.

Признак крана синхронизируется из полей вкладки "Колонна": `overheadCrane.enabled` или `suspendedCrane.enabled`. Если кран не включён, Summary не предлагает расчёт подкрановой балки, а старый результат подкрановой балки очищается.

Расчётные формулы колонн, generated-файлы и подбор профилей не изменялись.

## 2026-05-15 — вынесена общая модель количества ферм и балок покрытия

Добавлен helper `deriveRoofElementLayout` в `src/building/layout.ts`.

Он объединяет текущие правила:

- фермы покрытия считаются по внутренним рамам;
- балки покрытия на торцах считаются как `2 * spanCount`;
- существующие вкладки Truss и Beam Cell используют один общий источник количества.

Добавлены тесты для однопролётного и многопролётного режима. Поведение и numerical output не менялись: это централизация уже существующей логики.
