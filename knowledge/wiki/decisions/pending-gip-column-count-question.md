# Вопрос ГИПу — подсчёт крайних, средних и фахверковых колонн

Статус: `answered`.

Ответ зафиксирован в `knowledge/wiki/decisions/2026-05-15-gip-column-count-answer.md`.

Дата фиксации: 2026-05-15.

Связанные источники: `KM-004`, `KM-006`, `KM-009`, `SCN-BUILDING-COUNT-001`, `SCN-BUILDING-COUNT-002`, `SCN-BUILDING-COUNT-003`.

Связанный код: `src/building/layout.ts`, `src/building/unifiedLayout.ts`, `src/building/__fixtures__/building-count-scenarios.ts`, `src/SummaryApp.tsx`.

## Суть вопроса

В проекте сейчас есть два разных уровня модели подсчёта колонн.

Первый уровень — текущая рабочая логика приложения. Она считает основные крайние и средние колонны только по внутренним рамам, а торцы здания учитывает отдельно через фахверк.

Второй уровень — draft-модель по PDF-схемам VELICAN и сценариям `SCN-BUILDING-COUNT-001...003`. Она считает крайние и средние колонны по всем основным рамным осям `N`, включая торцевые оси.

Это не ошибка расчётной формулы колонны. Это вопрос классификации элементов здания для будущего раздела "Единое здание": считать ли торцевые колонны частью основных рам `edge/middle` или держать их отдельной группой `endFrame/fachwerk`.

## Текущая модель приложения

В текущем коде `deriveColumnLayout` использует:

```text
frameCount = floor(length / framePitch) + 1
interiorFrameCount = max(frameCount - 2, 0)
edge = 2 * interiorFrameCount
middle = max(S - 1, 0) * interiorFrameCount
```

Где:

- `S` — количество пролётов поперёк здания;
- `edge` — основные крайние колонны внутренних рам;
- `middle` — основные средние колонны внутренних рам;
- торцевые элементы идут отдельно через фахверк.

Следствие: торцевые оси `1` и `N` не попадают в группы `edge/middle`.

## Draft-модель по PDF/VELICAN

В предварительной модели по PDF-схемам рассматриваются все основные рамные оси:

```text
edge = 2 * N
middle = max(S - 1, 0) * N
total = (S + 1) * N
```

Где:

- `N` — количество основных рамных осей вдоль здания;
- торцевые оси `1` и `N` входят в общий подсчёт;
- промежуточные и дробные оси не смешиваются с основными рамными осями до отдельной проверки.

Эта модель пока не должна напрямую заменять текущую рабочую логику приложения.

## Пример расхождения

Сценарий: `SCN-BUILDING-COUNT-002`, источник `KM-006`, двухпролётная схема.

Исходные параметры для обсуждения:

```text
N = 20
S = 2
extraFrameAxes = 1
```

Текущая модель приложения, только внутренние рамы:

```text
interiorFrameCount = 18
edge = 2 * 18 = 36
middle = 1 * 18 = 18
total = 54
```

Draft-модель по всем основным рамным осям:

```text
edge = 2 * 20 = 40
middle = 1 * 20 = 20
total = 60
```

Разница относится к торцам:

```text
end edge = 4
end middle = 2
end total = 6
```

## Вопросы для ГИПа

1. Колонны на торцевых осях `1` и `N` считать обычными основными колоннами рам или отдельными торцевыми/фахверковыми элементами?
2. В многопролётной схеме средние колонны на торцевых осях считать основными средними колоннами или частью торцевого фахверка?
3. Если на торцевой оси стоит полноценная рама, ферма или балка покрытия, должна ли эта ось считаться обычной рамой для спецификации?
4. Как классифицировать дробные и дополнительные оси типа `А/1`, `Б/1`, `11/1`: фахверк, локальная стойка, дополнительная рама или отдельный тип элемента?
5. Для ведомости и будущей сводки нужны ли отдельные группы:
   - внутренние основные колонны;
   - торцевые основные колонны;
   - фахверковые стойки;
   - стойки ворот и проёмов;
   - стойки антресоли;
   - дополнительные локальные оси?

## Возможные решения

### Вариант A — оставить текущую модель

Основные `edge/middle` считаются только по внутренним рамам. Торцы остаются отдельной группой `fachwerk`.

Плюсы:

- соответствует текущему поведению приложения;
- минимальный риск для существующих расчётов;
- не смешивает основные рамы и торцевые элементы.

Минусы:

- количество основных колонн может казаться заниженным при сравнении с PDF-схемой, где все оси считаются вместе;
- для пользователя нужно явно объяснять, где находятся торцевые колонны.

### Вариант B — включить торцы в основные `edge/middle`

Основные колонны считаются по всем осям `N`. Фахверк содержит только дополнительные стойки, которые не совпадают с основными колоннами.

Плюсы:

- ближе к простой формуле `edge = 2 * N`, `middle = (S - 1) * N`;
- проще объяснить общий подсчёт по плану.

Минусы:

- может изменить текущую модель количества;
- нужно аккуратно исключить двойной счёт фахверковых стоек;
- потребуется пересмотреть Summary, тесты и будущую ведомость.

### Вариант C — хранить группы явно

Внутри модели здания считать отдельно:

```text
interiorFrameColumns
endFrameColumns
fachwerkPosts
extraAxisColumns
openingPosts
mezzaninePosts
```

А в UI и спецификации показывать как детальную разбивку и итог.

Плюсы:

- наиболее прозрачная модель для инженерной проверки;
- не прячет торцы;
- снижает риск двойного счёта;
- позволяет позже собрать нужную ведомость в формате ГИПа/КМ.

Минусы:

- требует чуть больше структуры данных;
- нужно согласовать термины с ГИПом.

Предварительная рекомендация для разработки: вариант C как внутренняя модель, но итоговое решение должен подтвердить ГИП.

## Что менять после ответа

Если ГИП подтверждает вариант A:

- оставить `deriveColumnLayout` без изменения;
- в Summary явно подписать, что `edge/middle` — только внутренние рамы;
- торцевые элементы показывать отдельной группой.

Если ГИП подтверждает вариант B:

- обновить модель количества в `deriveUnifiedBuildingLayout`;
- проверить влияние на будущую сводку массы и стоимости;
- обновить `SCN-BUILDING-COUNT-001...003`;
- отдельно проверить фахверк на двойной счёт.

Если ГИП подтверждает вариант C:

- развить `deriveUnifiedBuildingLayout` в явную модель групп;
- оставить расчётные формулы колонн без изменения;
- добавить acceptance-сценарии по одной односкатной, двухпролётной и четырёхпролётной схеме;
- показать в Summary отдельные строки по внутренним рамам, торцам и фахверку.

## Что не делать до ответа

- Не менять расчётные формулы колонн.
- Не менять numerical output существующих модулей.
- Не переносить draft-формулы из PDF напрямую в рабочий расчёт.
- Не объединять торцевые колонны и фахверк без инженерного решения.
- Не использовать диагностический блок Summary как окончательную ведомость металла.

## Короткая формулировка для отправки ГИПу

Нужно согласовать классификацию колонн на торцах здания. Сейчас в калькуляторе основные крайние и средние колонны считаются только по внутренним рамам, а торцы идут отдельной группой фахверка. В предварительной модели по схемам VELICAN можно считать основные колонны по всем рамным осям, включая торцы. Для двухпролётной схемы с `N = 20` это даёт разницу: текущая модель `36` крайних + `18` средних = `54`, модель по всем осям `40` крайних + `20` средних = `60`. Разница `6` колонн относится к торцевым осям. Нужно решить: торцевые колонны считать основными колоннами рам или отдельной группой фахверка/торцевых элементов?
